# PSM-Smart 实施路线图

本路线图按箱式变压器监测闭环组织，后续开发应优先保持当前可运行链路，再补充扩展能力。

## 第一阶段：数据库重建设计

目标：

- 使用箱变业务模型替换旧资产模型。
- 明确采样、告警、工单、柜门日志的数据关系。

任务：

1. 重写 `oracle21c-schema.sql`。
2. 创建 `BOX_TRANSFORMER`、`POWER_CIRCUIT`、`MEASURE_POINT`、`TS_RAW_DATA`、`CABINET_DOOR_LOG`、`ALARM_LOG`、`MAINT_TASK`、`SYS_USER`。
3. 为采样、告警、柜门日志和工单建立索引。
4. 使用显式序列维护主键。
5. 保留旧对象清理块，便于本地重建初始化。

验收：

- Oracle 21c 可执行初始化入口。
- 核心表、序列、外键、索引齐全。
- 采样表只保存固定 1 秒采样所需字段。

## 第二阶段：基础数据与演示数据

目标：

- 初始化可演示的箱变台账和采样闭环。

任务：

1. 初始化 1 台箱式变压器。
2. 初始化 1 个进线回路和至少 2 个出线回路。
3. 初始化完整测点字典。
4. 初始化四类角色用户。
5. 插入连续 1 秒采样数据。
6. 插入一个功率因数或温度越限告警。
7. 插入一条工单和一组柜门日志。

验收：

- 元数据层级完整。
- 演示数据可支撑前端历史、告警、工单和柜门事件展示。

## 第三阶段：后端元数据与查询

目标：

- 后端接口全面使用箱变、回路、测点参数。

任务：

1. 实现 `/api/metadata/transformers`。
2. 更新 `/api/history` 查询参数。
3. 更新 `/api/messages` 查询参数。
4. 更新 `/api/tasks` 查询参数。
5. 在 Service 层组装层级结构和默认时间范围。
6. 保持角色权限判断。

验收：

- 元数据接口返回箱变、回路、测点层级。
- 历史数据可按箱变、回路、测点查询。
- 消息查询覆盖采样、告警和工单。
- 工单查询和更新可用。

## 第四阶段：模拟采集与告警工单

目标：

- 系统能固定每 1 秒写入采样数据，并在异常时生成闭环数据。

任务：

1. 模拟采集定时任务固定 1 秒执行。
2. 每次采集读取启用测点。
3. 根据测点类型生成合理演示值。
4. 异常开关开启时生成越限或状态异常值。
5. 越限时写入 `ALARM_LOG`。
6. 严重告警写入 `MAINT_TASK`。
7. 状态接口返回写入数、告警数、工单数和采样间隔。

验收：

- 启动模拟后 `TS_RAW_DATA` 持续新增数据。
- 异常开关可生成告警。
- 严重告警可生成工单。
- 停止模拟后不再写入采样。

## 第五阶段：前端工作台

目标：

- 用户打开前端即可看到箱变监测工作台。

任务：

1. 封装 Dashboard API。
2. 定义箱变、回路、测点、历史数据、消息、工单和模拟状态类型。
3. 将筛选调整为箱变/回路/测点。
4. 实现历史数据聚合表和趋势图。
5. 实现告警和工单列表。
6. 实现工单反馈弹窗。
7. 实现 ADMIN 模拟测试和运行日志。

验收：

- `npm run build` 通过。
- 首屏没有模板残留。
- 筛选和数据列表联动正确。
- ADMIN 能启动/停止模拟并切换异常。

## 第六阶段：回归验证

目标：

- 确认全链路改动没有旧模型残留。

任务：

1. 后端执行 `mvn -q test`。
2. 前端执行 `npm run build`。
3. 扫描旧字段、旧接口和旧文案。
4. 在 Oracle 环境执行 `oracle21c-init.sql`。

验收：

- 后端测试通过。
- 前端构建通过。
- SQL 初始化通过。
- 代码和核心文档均使用箱变模型。
